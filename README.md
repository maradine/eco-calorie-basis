# Eco · Calorie Basis

A single-page app that computes the total labor calorie cost of any craftable
item in Eco, given your specific build choices. Built for sellers who want to
know what their goods cost them in labor so they can price accordingly.

Built from Strange Loop Games' autogenerated recipe source. **1,370 recipes
producing 1,221 distinct items, fully indexed.**

## What it does

Pick any craftable item. See:

- **Calories per unit** — sum of all labor burned to produce one, recursively down to raw inputs
- **Order total** — same figure scaled by your batch quantity
- **Breakdown tree** — collapsible view of every sub-recipe, ingredient, and raw harvest
- **Raw harvest requirements** — flat list of "what to go gather"
- **Decisions to pin** — every ambiguity in the dependency graph, whether on the current path or not, so you can lock in your actual build before seeing a number

## How it resolves ambiguity

Two kinds of ambiguity show up in Eco recipes:

- **Tag ingredients**: a recipe says "needs Wood" and any log satisfies it
- **Multiple recipes**: an item can be produced by several different recipes (e.g. BoardItem via Board, Boards, SawBoards, or ParticleBoards)

**Defaults**: alphabetical-first by ID. Stable, obvious, no clever optimization. When you need a Board, the default recipe is `Board`, not `SawBoards`, even though SawBoards is cheaper per unit — because this is a seller's tool and your build is whatever you choose it to be, not whatever an optimizer says it should be.

**Overrides**: sticky. Once you pin `tag:WoodBoard → HardwoodBoardItem` or `item:BoardItem → SawBoards`, that's the answer. The resolver will not silently reroute around your choice to find a cheaper path.

**Off-path decisions**: surfaced anyway. If a change to one override would bring another item or tag into play, you still see all the alternate branches in the overrides panel — they're just dimmed. This lets you see the full topology of decisions available to you before committing.

```sh

npm run build     # production build to dist/
npm run preview   # serve the built version locally
```

Deploy `dist/` to any static host (GitHub Pages, Firebase Hosting, Netlify, etc.).

## Assumptions & caveats

- **Level 0, no upgrades.** Every recipe uses its base labor cost. Skill levels, talents, and upgrade modules are not modeled. This gives a baseline — actual labor burned will be lower with skills and upgrades.
- **Raw harvest = 20 cal / unit.** Every item with no producing recipe is treated as "one axe/pickaxe swing = 20 cal = 1 unit." This matches the in-game model where every harvesting tool burns 20 cal/swing and appropriate-tier tools break blocks in one hit. Override per-item via the `rawCosts` option on the `Resolver` constructor if your setup differs.
- **Multi-output recipes allocate labor across all outputs.** For a recipe like `ButcherBison` (10 meat + 2 hide + 3 wool), the labor is divided by the output quantity being priced. So one unit of meat costs labor/10, not the whole 70. This may or may not match how you think about cost allocation — if not, the place to adjust is `resolveRecipe` in `resolver.ts`.

## Project layout

```
public/
  eco-data.json          pre-bundled recipe + tag data
scripts/
  parse_recipes.py       extract recipes from Eco's autogen .cs files
  build_tags.py          extract [Tag("x")] memberships from *Item classes
  build_bundle.py        combine both into public/eco-data.json
src/
  App.tsx                top-level app
  styles.css             styling (field-report aesthetic)
  components/
    Picker.tsx           searchable item selector + qty
    Totals.tsx           headline stats
    Tree.tsx             collapsible breakdown tree
    Overrides.tsx        ambiguity override controls
    FlatInputs.tsx       flat raw-harvest list
  lib/
    resolver.ts          core graph walker (memoized DFS + cycle detection)
    types.ts             data schema
    smoketest.ts         regression tests (tsx src/lib/smoketest.ts)
```

## Regenerating the data

If you update to a newer Eco autogen source tree, the `scripts/` directory has
Python scripts that walk it:

```sh
python3 scripts/parse_recipes.py    # produces scripts/recipes.json
python3 scripts/build_tags.py       # produces scripts/tags.json
python3 scripts/build_bundle.py     # combines both -> public/eco-data.json
```

The parse scripts currently hardcode the path to Eco's source tree at the
top of each script — adjust to match your local layout.

The scripts were tested against a specific Eco autogen snapshot. If the
recipe grammar changes, the regexes at the top of `parse_recipes.py` are
the places to look first.
